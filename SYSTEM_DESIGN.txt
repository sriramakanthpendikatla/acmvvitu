1. Project Overview
ACM VVITU is the official website for the Association for Computing Machinery chapter at VVIT University. It serves as
the public face of the chapter for students, faculty, and recruiters, while providing a private admin panel for chapter
officers to manage all site content without touching code.

Vercel (auto-deploy from GitHub main branch)

acm.vvit.edu.in (or equivalent institutional domain)
Goals
   Showcase ACM VVITU events, team, and projects to students and recruiters.
   Provide a zero-code admin panel so officers can update content independently.
   Deliver a premium first impression with the existing 3D Three.js hero scene.
   Support event registration with automatic confirmation emails.
   Be fully responsive across desktop and mobile devices.

3. High-Level Architecture
The system follows a three-tier architecture entirely within the Next.js monorepo. There is no separate backend server

4. Module Breakdown
HERO MODULE
   Wraps scene.js + index.html as a React component (SceneCanvas.tsx).
   Loaded with next/dynamic and ssr:false — Three.js needs the browser window.
   useScene.ts hook manages renderer lifecycle, resize observer, and cleanup.
   Intro sequence (loading bar → letter drop → constellation trace) preserved.
   Mouse parallax and responsive camera fit work identically to the standalone version.
EVENTS MODULE
   EventCard component: banner image, title, date, venue, status badge, register button.
   Timeline view: events sorted by date, grouped by upcoming / past.
   Registration button links to the embedded custom form (not Google Form).
   Admin: create / edit / delete events, upload banner image, set registration open/closed.
TEAM MODULE
   MemberCard: photo, name, designation, year, LinkedIn + GitHub icons.
   Filterable by year and is_core (core committee vs general member).
   Photos stored in Supabase Storage bucket; URLs saved in team_members table.
   Admin (admin role only): add/remove members, update roles and photos.
PROJECTS MODULE
   ProjectCard: cover image, title, tags (tech stack), description, repo link.
   Filterable by tag.
   Admin (editor+): add/edit/delete projects, upload cover image.
ADMIN PANEL
   Route: /admin/* — protected by middleware.ts + server-side role check.
   Sidebar nav: Dashboard, Events, Team, Projects, Registrations, Settings, Users (admin only).
   DataTable component: sortable, searchable, paginated — reused across all entities.
   EventForm / TeamForm / ProjectForm: Zod-validated, image upload via Supabase Storage.
   Registrations view: read-only table with CSV export for admins.
   User management (admin only): invite by email, assign role (admin or editor), revoke access.
AUTH MODULE
   NextAuth.js v5 with Credentials provider (email + password).
   JWT session includes id, email, role claim.
   middleware.ts: blocks /admin/* if no valid session; redirects to /admin/login.
   Each API route re-validates role from session before mutating data (defense-in-depth).
   Supabase Row Level Security adds a third enforcement layer directly at the DB.

6. Role-Based Access Control
Three principals interact with the system: unauthenticated public visitors, authenticated editors (limited write access),
and admins (full control). Permissions are enforced at three independent layers: Next.js middleware (route guard), API
route handlers (role check from session JWT), and Supabase RLS policies (database level).

Role assignment flow
   Admin creates a new user account from the /admin/users panel.
   User receives an invitation email (via Resend) with a one-time setup link.
   User sets their password; NextAuth creates a session with the role claim from the DB.
   Subsequent logins read the role from the users table on each session refresh.
   Admin can upgrade editor → admin or downgrade admin → editor at any time.
   Revoking access: set the user record to inactive; middleware rejects their session.

7. Key Interaction Flows
7.1 Admin login and role routing
Visit /admin
middleware.ts
Session?
Role check
Dashboard
   No session → redirect to /admin/login → NextAuth Credentials provider verifies email + password.
   Valid session, role=admin → full dashboard (all tabs including Users).
   Valid session, role=editor → limited dashboard (Events + Projects only; Users tab hidden).
   Invalid role (corrupted JWT) → 403 response; middleware logs the attempt.
7.2 Event registration (public visitor)
Click Register
Fill form
Zod validate
POST /api/register
Write DB + Email
Success
   Form fields: full name, roll number, branch, email address.
   Client-side Zod validation fires before the request is sent — no round-trip for obvious errors.
   API route re-validates with the same Zod schema (defense-in-depth).
   On success: row inserted in registrations table, Resend fires confirmation email to the student.
   On duplicate (same email + event_id): return 409 with friendly message — no double entries.
7.3 Content update (editor workflow)
Login /admin
Open Events
Edit/Create
Upload banner
Save → API
Published
   Image upload: file sent to /api/upload → stored in Supabase Storage bucket → public CDN URL returned.
   Form submit: Zod-validated payload sent to /api/events (POST for create, PATCH for edit).
   API route checks session role before writing — editors cannot update team or users.
   Supabase RLS double-checks the claim from the session token.
   On success: ISR revalidation tag fires so the public page reflects the change within seconds.

11. Development Roadmap
PHASE 1 — FOUNDATION (WEEK 1–2)
   Next.js project scaffolding with App Router, Tailwind, shadcn/ui.
   Supabase project setup: create all 6 tables, enable RLS, create storage bucket.
   NextAuth integration: Credentials provider, JWT role claim, session types.
   middleware.ts route guard for /admin/*.
   Navbar + Footer components.
PHASE 2 — HERO INTEGRATION (WEEK 2–3)
   Refactor scene.js into SceneCanvas.tsx + useScene.ts hook.
   Dynamic import with ssr:false; loading fallback matches original loading screen.
   HeroSection.tsx wrapping the canvas with the existing HUD overlay.
   Responsive camera and mobile scaling verified on real devices.
PHASE 3 — PUBLIC SECTIONS (WEEK 3–4)
   AboutSection: static content from site_meta table.
   EventsSection: fetch from /api/events, EventCard + timeline layout.
   TeamSection: fetch from /api/team, MemberCard + year/core filter.
   ProjectsSection: fetch from /api/projects, ProjectCard + tag filter.
   ContactSection: social links, WhatsApp link, contact email from site_meta.
PHASE 4 — REGISTRATION FORM (WEEK 4)
   Registration form component with Zod validation.
   POST /api/register: write to DB, send confirmation via Resend.
   Duplicate detection (same email + event).
   Success/error toast feedback.
PHASE 5 — ADMIN PANEL (WEEK 5–6)
   Admin layout with sidebar navigation and role-aware tab visibility.
   DataTable component (sort, search, paginate) — reused across all entities.
   EventForm, TeamForm, ProjectForm with image upload to Supabase Storage.
   Registrations view with CSV export.
   User management (admin only): invite, assign role, revoke access.
   site_meta editor: update About text, contact email, social links.
PHASE 6 — POLISH & LAUNCH (WEEK 7)
   Mobile responsiveness audit and fixes.
   Lighthouse performance pass: lazy-load images, ISR revalidation tags.
   SEO: meta tags, Open Graph image, robots.txt, sitemap.
   Error boundaries and 404 / 500 pages.
   Final security review: CSP headers, Supabase RLS audit.
   DNS → custom domain on Vercel, SSL certificate auto-provision.
